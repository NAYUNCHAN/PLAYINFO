# KRA API 한계 보고서 초안

## 1. PoC 목적

Race Note v1에서 무료 인프라로 안정적으로 수집·정규화할 수 있는 KRA 데이터 범위를
실제 호출과 표본 비교로 결정합니다.

## 2. 확인 대상 API

경주정보, 경주결과, 확정배당, 마체중, 출전취소, 기수변경을 대상으로 합니다.

## 3. 실제 호출 가능 여부

현재 작업 환경에는 `KRA_API_KEY`와 `KRA_API_BASE_URL` 실제 값이 없어 호출을 수행하지
못했습니다. CLI는 누락된 환경변수 이름만 출력하고 종료하도록 준비했습니다.

## 4. API별 제공 필드 목록 요약

공식 설명에서 경주·말·기수·조교사·결과·배당·마체중 관련 필드를 확인했습니다.
정확한 JSON 필드명과 타입은 [field-mapping.md](./field-mapping.md)에 추정값으로 기록했습니다.

## 5. 누락 필드 목록

식별자 일관성, 확정 상태, 변경 전후 번호 등은 실제 샘플 대조가 필요합니다.
[missing-fields.md](./missing-fields.md)를 기준으로 조사합니다.

## 6. 호출 제한 및 실패 가능성

개발계정 호출량, 활용신청 승인, endpoint 변경, JSON/XML 형식 차이, 일시적 서버 오류가
주요 실패 요인입니다. retry는 일시 장애만 완화하며 인증·명세 오류를 해결하지 않습니다.

## 7. 데이터 반영 지연 가능성

공식 페이지의 업데이트 주기 표기만으로 개별 필드의 반영 시점을 보장할 수 없습니다.
경주 전후 일정 간격으로 호출해 최초 관측 시각과 값 변경 시각을 기록해야 합니다.

## 8. 출전취소 제공 여부

공공데이터포털에서 전용 데이터 상품과 사유 제공 설명을 확인했습니다. 승인된 요청주소,
실제 필드명, 경마장 범위와 반영 시점은 보류 상태입니다.

## 9. 기수변경 제공 여부

공공데이터포털에서 전용 데이터 상품과 변경 전후 기수명·부담중량 제공 설명을 확인했습니다.
기수번호 제공 여부와 승인된 요청주소는 보류 상태입니다.

## 10. 결과/배당 반영 시점 확인 계획

같은 경주를 종료 직후부터 간격을 두고 조회해 결과 확정 표식, 착순, 승식별 배당의 최초
제공 및 마지막 변경 시각을 기록합니다. 화면에는 예정 시각이 아니라 마지막 성공 시각과
공식 결과 대기 여부를 표시하는 제품 원칙을 유지합니다.

## 11. v1에서 자동화 가능한 데이터

안정적인 식별자와 확정 상태가 확인된 경주 일정, 출전 정보, 결과, 마체중 및 배당의
정규화 필드가 후보입니다. 실제 PoC 통과 전에는 자동화 범위를 확정하지 않습니다.

## 12. v1에서 수동 처리해야 하는 데이터

식별자가 없거나 충돌한 말·기수·조교사 연결, 제공되지 않는 변경 사유, 공식 확정 상태를
판별할 수 없는 데이터는 `needs_review`로 분류해 수동 확인합니다.

## 13. Supabase 정규화 저장 전제

필요한 필드만 저장하고 원본 응답 전문은 운영 DB에 보관하지 않습니다. Service Role Key는
서버 전용 수집기에서만 사용하고 브라우저 코드와 로그에 노출하지 않습니다.

## 14. upsert 검증 필요 항목

unique 후보 충돌, 재실행 멱등성, null 덮어쓰기, 확정 전후 변경, 배당 조합 정규화,
경마장별 식별자 범위를 검증해야 합니다.

## 15. PoC 통과/보류 판정 기준

성공 응답 재현, 필드 타입 확인, 식별자 일관성, 확정 상태 판별, 무료 호출량 내 수집이
확인되면 통과합니다. 핵심 식별자 또는 확정 상태가 불명확하면 해당 API 자동화는 보류합니다.

## 16. 남은 리스크

- 일부 공식 명세가 XML로만 표시되어 JSON 요청 지원 여부가 다를 수 있음
- 상품별 요청 변수명과 코드 체계가 다를 수 있음
- 말·기수·조교사 번호가 API 및 경마장 사이에서 동일하지 않을 수 있음
- 취소·변경·결과·배당의 반영 시점을 보장할 수 없음
- 개발계정 호출량 안에서 필요한 재조회 횟수를 충족하지 못할 수 있음
